Design Justification
## Storage Systems
(Which databases/storage systems did you choose for each of the 4 goals and why)
For the patient readmission prediction model (Goal 1), I selected Amazon S3 for storing historical treatment data and trained ML models, combined with PostgreSQL as the operational database for structured patient records. PostgreSQL provides ACID compliance for critical healthcare data while S3 offers cost-effective storage for large historical datasets and model artifacts. The ML training pipeline reads PostgreSQL, processes data, and stores models in S3 for versioning.
For natural language querying (Goal 2), I implemented a vector database (Pinecone or pgvector extension in PostgreSQL) to store embedded patient history alongside the primary PostgreSQL database. This enables semantic search capabilities where doctors' natural language questions are converted to embeddings and matched against indexed patient records, providing fast retrieval of relevant medical history.
For monthly management reports (Goal 3), I chose Amazon Redshift as the data warehouse. Patient data, billing records, and operational metrics are ETL'd nightly from PostgreSQL into Redshift's columnar storage, optimized for complex analytical queries across departments, time periods, and cost centers. This OLAP system handles aggregations efficiently without impacting transactional systems.
For real-time ICU vitals (Goal 4), I implemented Amazon Kinesis Data Streams for ingestion, Amazon Timestream for time-series storage, and Redis for caching recent vitals. Kinesis handles high-throughput streaming from monitoring devices, Timestream provides optimized storage and querying for time-series data, and Redis enables sub-second access to current patient vitals for clinical dashboards.
## OLTP vs OLAP Boundary
(Where does the transactional system end and the analytical system begin in your design)
The OLTP system consists of PostgreSQL handling real-time patient admissions, treatment records, medication orders, and billing transactions. This transactional layer prioritizes data consistency, concurrent writes, and low-latency reads for clinical workflows. The boundary occurs at the nightly ETL pipeline that extracts data from PostgreSQL and loads it into Redshift.
The OLAP system begins with Redshift, where historical data is denormalized, aggregated, and optimized for analytical queries. Management reports, trend analysis, and business intelligence tools query Redshift exclusively, ensuring that complex analytical workloads don't impact clinical operations. The ICU vitals streaming pipeline operates parallel to this boundary—Kinesis and Timestream handle real-time analytics independently, while summary statistics are periodically pushed to Redshift for historical reporting.
This clear separation ensures that doctors and nurses experience consistent performance during patient care, while analysts and administrators can run resource-intensive queries without affecting clinical systems. The ML training pipeline reads from both systems: recent data from PostgreSQL for model updates and historical data from Redshift for comprehensive training.
## Trade-offs
(Identify one significant trade-off in your design and how you would mitigate it)
A significant trade-off in this design is data freshness versus system performance. Management reports in Redshift are updated nightly, meaning executives see data that's up to 24 hours old. For rapidly changing metrics like bed occupancy, this latency could impact operational decisions during surge events or capacity planning.
To mitigate this, I would implement a hybrid approach: maintain the nightly ETL for comprehensive reports, but add a real-time dashboard layer using materialized views in PostgreSQL or a streaming analytics service (Amazon Kinesis Data Analytics) that provides live metrics for critical operational KPIs. This dashboard would show current bed occupancy, ER wait times, and department utilization by querying PostgreSQL replicas or streaming aggregates, while detailed historical analysis remains in Redshift. This balances the need for real-time operational visibility with the cost and complexity of maintaining a fully real-time data warehouse.
